## 1. システム概要

### 1.1 成果物
**1つの予約専用サイト**（三田市クリーンセンター）を構築する。

```
[三田市役所サイト（既存）] ──link──> [本サイト（本案件）]
```

- サイト全体が「ごみ持込予約」のための専用サイト
- 公式HP = 予約サイト（1つのアプリ）
- 三田市役所の本来のサイトは別途存在（既存・本案件範囲外）

### 1.2 技術スタック（確定）

| 項目 | 採用 |
|---|---|
| 言語 | PHP 8.2+ |
| FW | Laravel 11 (LTS) |
| DB | MySQL 8.x |
| インフラ | Xserver VPS（メーカ管理・24/365監視） |
| フロント | SCSS + Vanilla JS（必要時 Alpine.js） |
| メール | SendGrid + SPF/DKIM/DMARC |
| Bot対策 | reCAPTCHA v3 |
| QR | endroid/qr-code |
| 監査ログ | spatie/laravel-activitylog |
| CSV | league/csv |

### 1.3 サイト構成

```
持込予約サイト（Laravel 1つのアプリ）
├── /              トップ（施設案内・利用ガイド・注意事項・同意）

└── /admin         管理画面
      ├── ログイン
      ├── ダッシュボード
      ├── 予約管理
      ├── 受付枠設定
      ├── 品目マスタ
      ├── お知らせ管理
      ├── 管理者設定
      └── ログ閲覧
```

## 2. 認証設計（確定）

### 2.1 利用者側: 会員登録なし

**方針**: 会員登録なし。予約番号 + 電話番号認証を基本とする。

根拠:
- 仕様書11「メールアドレスを持たない人も簡単に予約変更等できること」
- 設計書「会員登録不要で簡単に予約可能」
- ごみ持込は低頻度サービス（年数回）、会員登録のメリットが薄い
- 高齢者を含む全市民が対象、ID/PW管理が壁になる
- 個人情報の長期保管リスクを最小化

### 2.2 利用者の認証フロー

```
【予約確認・変更・取消】

パターンA: メール持ちユーザー（主な導線）
確認メール内のマジックリンク
    ↓（ワンタッチ）
予約内容表示
    ↓
変更・取消ボタン押下
    ↓
電話番号入力（本人確認）
    ↓
変更・取消実行

パターンB: メール無しユーザー（仕様書11対応）
マイページ（/mypage）
    ↓
受付番号 + 電話番号 入力
    ↓
予約内容表示 → 変更・取消実行
```

### 2.3 受付番号を忘れた場合

**施設に電話 → 受付員が代理対応**（仕様書12の代理入力機能で吸収）。

- システム側で「番号忘れ」の自己解決手段は設けない
- 注意事項ページに「受付番号は大切に保管してください。不明な場合は施設にお電話ください」と表示
- 受付員が管理画面で電話番号 or 氏名で検索し、代理で変更・取消対応

理由:
- 電話番号だけで検索できると本人確認が弱い（他人が取消可能になる）
- 仕様書12の代理入力機能がまさにこのケースを想定している

### 2.4 マジックリンクの仕様

- URLにトークン（推測不可能な長い文字列）を埋め込む
- 有効期限: **持込日まで**（持込日翌日に無効化）
- 使い捨てではない（何度でもアクセス可能）
- 変更・取消には追加で電話番号入力が必要

### 2.5 QRコードの用途

- **主目的: 現地受付での予約照合**（施設側がスキャン）
- 受付員がスキャン → 管理画面で予約情報を参照
- 計量システムとの連携にも使用
- マジックリンクとは**別の用途**（変更画面への遷移には使わない）
- QRには受付番号をエンコード（URLではなくテキスト）

### 2.6 2回目以降の入力省略（仕様書10）

**電話番号で過去予約情報を呼び出す**方式:

```
新規予約フォーム
    ↓
電話番号入力
    ↓
過去に同じ電話番号で予約あり？
    ├─ Yes → 氏名・住所等を自動入力（編集可）
    └─ No  → 通常入力
```

- 明示的なログイン不要
- パスワード不要
- 設計書「会員登録不要」と整合

### 2.7 管理者側: ログイン必須

- ID / パスワード認証
- Figmaでログイン画面が設計済み（`85:1329`）
- 「ログイン状態を保持する」チェックボックスあり
- パスワード忘れはシステム管理者に問い合わせ

## 4. 予約フロー

### 4.1 利用者フロー（7画面）

```
TOP（/）
└── 利用ガイド / 注意事項 / 利用規約同意
    ↓ 同意して予約開始
予約フォーム（/form）
└── Step1: 利用者情報（氏名・連絡先・住所）
└── Step2: 日時選択（カレンダー・時間帯）
└── Step3: 品目選択（分類/50音/キーワード検索）
    ↓
内容確認（/confirm）
└── 全入力内容の確認・修正
    ↓
予約完了（/done）
└── 受付番号 + QRコード表示
└── 確認メール自動送信（メールあり利用者）

マイページ（/mypage）
└── 受付番号 + 電話番号 入力（認証）
└── 予約内容表示
└── 変更 or 取消
```

### 4.2 変更できる項目（仕様書基本6）

- 品目の追加・取消
- 持込日
- 持込時間帯

### 4.3 受付枠制御

- 上限到達した日・時間帯はカレンダーで選択不可（グレーアウト）
- 一般/事業系で**別々に**上限管理（仕様書基本7）
- 1時間単位で時間帯選択

## 5. 受付番号・QRコード（仕様書基本15）

- 受付完了時に受付番号とQRコードを発行
- 受付番号体系: **未確定**（例: `R-0513-001` がFigmaのサンプル）
- リセット単位: 年/月/週 で管理画面から設定可能
- QRコードには受付番号をエンコード（URL形式ではない）
- 確認メールに受付番号・QRコードを記載（仕様書基本16）

## 6. メール通知（仕様書基本8・16・17）

| トリガー | 送信先 | 内容 |
|---|---|---|
| 新規予約完了 | 利用者（メールあり） | 受付内容・受付番号・QRコード・マジックリンク |
| 予約変更 | 利用者（メールあり） | 変更後の内容 |
| 予約取消 | 利用者（メールあり） | 取消の確認 |
| 持込前日 | 利用者（メールあり） | リマインド（cron で自動送信） |

- メールアドレス未登録の利用者にはメール送信しない（エラーにもしない）
- 送信: SendGrid（外部SMTP）+ SPF/DKIM/DMARC

## 7. 管理画面機能

### 7.1 Figmaで確認済みの画面（`node-id=1:4`）

| 画面名 | 主な機能 |
|---|---|
| ログイン | ID/PW認証 |
| ダッシュボード | 本日の予約数・空き枠・未確認件数・今月合計 |
| 予約管理 | 一覧・検索・CSV出力・詳細 |
| 予約詳細 | ステータス変更・内部メモ・操作ログ |

### 7.2 Figmaにメニューはあるが画面未設計

- **受付枠設定**: 一括設定（施設・期間・曜日）+ 個別設定（1日ごと・時間帯ごと）+ 家庭/事業別管理
- **お知らせ管理**: 注意事項の編集（日英両言語）

### 7.3 Figmaに存在しない（追加実装必要）

- **品目マスタ**: 分類・名称・手数料・適用月・持込可否の編集
- **代理入力フォーム**: 受付員が利用者に代わって予約登録・変更・取消
- **ログ閲覧**: ログイン履歴・個人情報アクセス監査ログ
- **受付番号リセット設定**: 年/月/週の切替
- **管理者設定**: 権限管理等

### 7.4 代理入力機能（仕様書基本12）

- TEL受付員が利用者に代わって入力
- 当日持込（原則禁止）の現場登録にも使用
- 代理入力時は確認メール送信のON/OFF切替可能
- 代理入力者が後から変更・取消も可能
- 受付員ログイン済みの管理画面から操作

## 8. 外部連携

### 8.1 計量システム連携（仕様書管理4）

- 計量システムは**他社・範囲外**
- CSV形式で受付情報を出力・取込
- CSV仕様（文字コード・列定義・フォーマット）は**計量システム側の指定に従う（未確定）**
- 予約外搬入者への警告データも提供

### 8.2 郵便番号API

- 郵便番号入力で住所自動補完（仕様書基本9）

## 9. セキュリティ
